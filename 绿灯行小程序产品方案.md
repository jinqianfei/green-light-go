# 「绿灯行」小程序产品方案

> 盲人过马路红绿灯识别助手 — 产品设计文档 v1.2（联合评审修订版）

## 1. 产品概述

### 1.1 产品名称
**绿灯行** — 一句话说明：按住按钮，AI帮你"看"红绿灯。

### 1.2 产品定位
专为视障人士设计的**极简红绿灯识别工具**。用户只需按住屏幕一个大按钮，AI通过手机摄像头实时识别红绿灯状态，并用语音播报结果。

### 1.3 核心价值
- **安全**：解决盲人过马路时无法感知红绿灯的致命痛点
- **极简**：一个按钮完成所有操作，零学习成本
- **轻量**：微信小程序，无需安装，打开即用

### 1.4 目标用户
- 全盲或低视力人士
- 使用手机读屏软件（TalkBack/VoiceOver）的用户
- 日常需要独立过马路的视障人群

## 2. 功能设计

### 2.1 功能清单（MVP）

| 功能 | 描述 | 优先级 |
|------|------|--------|
| **红绿灯识别** | AI实时识别摄像头画面中的红绿灯状态（红/绿/黄） | P0 必须 |
| **语音播报** | 用语音告知用户当前红绿灯状态 | P0 必须 |
| **一键操作** | 按住屏幕按钮开始识别，松开停止 | P0 必须 |
| **振动反馈** | 识别到不同状态时给出不同振动模式（绿=短振、红=长振） | P0 必须 |
| **置信度提示** | 当AI不确定时，播报"无法确认，请小心" | P0 必须 |
| **使用引导** | 首次打开时的语音引导，教用户如何使用 | P0 必须 |
| **音量检测** | 提醒用户调高手机音量，确保能听到播报 | P1 重要 |
| **免责声明** | 首次使用时语音播报免责提示 | P0 必须 |

### 2.2 明确不做（MVP阶段）

- ❌ 不做路线规划和导航
- ❌ 不做避障功能
- ❌ 不做志愿者求助
- ❌ 不做用户账号系统
- ❌ 不做付费功能

### 3.4 持续识别与信号变化处理

用户持续按住按钮时，AI以**每2秒**为间隔持续识别（避免频繁播报干扰用户）。

**信号变化播报策略（防抖机制）：**

| 规则 | 说明 |
|------|------|
| 同色不重复播报 | 连续识别到相同颜色，仅在首次播报，后续静默 |
| 变色立即播报 | 检测到颜色变化时，立即播报新状态（不受2秒间隔限制） |
| 低置信度→高置信度 | 从"不确定"变为"确定"时，播报确定结果 |
| 高置信度→低置信度 | 从"确定"变为"不确定"时，播报"信号不清晰，请确认" |
| 静默超时 | 连续10秒无变化，每10秒播报一次当前状态（防止用户忘记） |

**示例场景：**
```
用户按住 → "正在识别" → "绿灯，可以通行"
（2秒后继续识别，仍为绿灯，静默）
（4秒后继续识别，变为红灯）
→ 立即播报 "注意，已变红灯"
（6秒后仍为红灯，静默）
（14秒后仍为红灯，10秒超时）
→ 播报 "当前红灯"
```

### 3.5 语音播报规则

| 场景 | 播报内容 | 振动模式 |
|------|---------|---------|
| 按住开始 | "正在识别" | 短振1次 |
| 识别到绿灯 | "绿灯，可以通行" | 短振1次 |
| 识别到红灯 | "红灯，请等待" | 长振1次 |
| 识别到黄灯 | "黄灯，请注意" | 中振2次 |
| 无法识别 | "无法确认，请小心通过" | 短振2次 |
| 信号变化（绿→红） | "注意，已变红灯" | 长振2次 |
| 信号变化（红→绿） | "已变绿灯，可以通行" | 短振2次 |
| 首次打开 | 语音引导（见上方） | 无 |

### 3.5 读屏软件适配

| 适配项 | 说明 |
|--------|------|
| 所有文字元素添加 `aria-label` | 确保TalkBack能朗读 |

## 4. 技术架构

### 4.1 整体架构

```
┌─────────────────────────────────────────┐
│              微信小程序前端               │
│  ┌─────────┐ ┌──────────┐ ┌──────────┐  │
│  │ 摄像头   │ │ 语音播报  │ │ 振动反馈  │  │
│  │ 组件     │ │ (TTS)    │ │ (Vibrate)│  │
│  └────┬────┘ └──────────┘ └──────────┘  │
│       │                                  │
│  ┌────┴────────────────────────────┐     │
│  │     AI推理（端侧/云侧混合）       │     │
│  └────────────┬───────────────────┘     │
└───────────────┼─────────────────────────┘
                │
        ┌───────┴────────┐
        │                │
   ┌────┴────┐    ┌──────┴──────┐
   │ 端侧推理 │    │  云端推理    │
   │ (优先)   │    │ (降级方案)   │
   │ 小模型   │    │ 大模型+GPU   │
   └─────────┘    └─────────────┘
```

### 4.2 技术选型

| 模块 | 技术方案 | 说明 |
|------|---------|------|
| **前端框架** | 微信小程序原生 / uni-app | MVP建议原生开发，性能更好 |
| **AI模型** | YOLOv8n（nano） | 轻量级目标检测，适合移动端 |
| **端侧推理** | 微信小程序WNN插件（优先）/ 云端降级 | ⚠️ WNN插件目前仅部分安卓机型支持，需做好云端降级 |
| **云端推理** | Flask + ONNX Runtime / TensorRT | 端侧不支持时自动降级到云端 |
| **语音播报** | 微信 `wx.createInnerAudioContext` + 预录制音频 | 比TTS更稳定、延迟更低 |
| **振动反馈** | 微信 `wx.vibrateLong` / `wx.vibrateShort` | 微信原生API |
| **后端服务** | 微信云开发 / 轻量云服务器 | 仅需模型推理服务 |
| **模型训练** | Python + PyTorch + Ultralytics | YOLOv8官方框架 |

### 4.2.1 端侧推理兼容性说明（重要）

微信小程序的端侧AI推理能力目前有限，需要做好分级策略：

| 方案 | 支持情况 | 说明 |
|------|---------|------|
| **WNN（WebNN）插件** | 仅部分安卓机型（2024年起逐步开放） | 微信官方推荐，但覆盖不全 |
| **TensorFlow.js 小程序插件** | 兼容性一般，包体积大 | 不推荐作为主方案 |
| **云端推理（主方案）** | 所有设备均支持 | **MVP阶段建议以云端推理为主** |

**MVP推荐策略：先做云端推理，保证所有设备可用；后续再逐步接入端侧推理降低延迟和服务器成本。**

```
MVP阶段（优先保证可用性）：
  所有设备 → 云端推理（200-500ms延迟）

V2阶段（优化体验）：
  支持WNN的安卓设备 → 端侧推理（<100ms）
  其他设备 → 云端推理（降级）
```

### 4.3 AI模型方案

#### 模型选择
- **基础模型**：YOLOv8n（nano版本）
  - 模型大小：~6MB
  - 推理速度：手机端 >15FPS
  - 参数量：3.2M

#### 训练数据集
需要采集和标注**中国城市红绿灯**的专用数据集，采用**开源数据集 + 自采数据**组合策略降低成本：

**开源数据集（免费，直接使用）：**

| 数据集 | 规模 | 来源 | 说明 |
|--------|------|------|------|
| **TT100K（Tsinghua-Tencent 100K）** | 100,000张 | 清华-腾讯 | 含交通标志和信号灯，中国场景 |
| **Bosch Small Traffic Lights Dataset** | 24,000张 | 博世 | 德国场景，可迁移学习 |
| **LISA Traffic Light Dataset** | 43,000帧 | UCSD | 美国场景，含昼夜标注 |
| **ROD (Real-time Object Detection) Dataset** | 10,000+张 | 开源 | 含交通灯类别 |

**自采数据（补充中国特有样式）：**

| 数据类别 | 数量目标 | 说明 |
|---------|---------|------|
| 中国横排灯 | 500+张 | 最常见样式 |
| 中国竖排灯 | 500+张 | 常见样式 |
| 带倒计时LED灯 | 300+张 | 新型灯 |
| 箭头灯（左转/右转/直行） | 300+张 | 复杂路口 |
| 夜间场景 | 300+张 | 低光照条件 |
| 恶劣天气（雨/雾） | 200+张 | 边缘场景 |
| **自采总计** | **~2,100张** | |

**数据策略：先用开源数据集预训练（低成本），再用自采数据微调（提升中国场景准确率）。总标注成本可控制在5,000-10,000元以内。**

#### 模型输出
```
检测结果 = {
  "light_detected": true/false,     // 是否检测到红绿灯
  "color": "red"|"green"|"yellow",  // 灯的颜色
  "confidence": 0.85,               // 置信度
  "direction": "straight"|null,      // 箭头方向（MVP可不做）
  "countdown": null                  // 倒计时（MVP不做）
}
```

#### 置信度策略
| 置信度 | 行为 |
|--------|------|
| ≥ 0.8 | 正常播报识别结果 |
| 0.5 - 0.8 | 播报结果 + "建议确认" |
| < 0.5 | 播报"无法确认，请小心" |

### 4.4 端侧 vs 云端推理策略

```
用户按住按钮
    │
    ├── 检测手机性能（NPU是否可用）
    │       │
    │   ┌───┴───┐
    │   │       │
    │ 支持端侧  不支持端侧
    │   │       │
    │ 端侧推理  云端推理
    │ (优先)   (降级)
    │   │       │
    │   └───┬───┘
    │       │
    │   返回识别结果
    │       │
    └─── 语音播报 + 振动反馈
```

| 方案 | 延迟 | 依赖网络 | 覆盖设备 | 成本 |
|------|------|---------|---------|------|
| 端侧推理 | <100ms | 不需要 | 仅支持WNN的设备 | 零 |
| 云端推理 | 200-500ms | 需要 | 所有设备 | 服务器成本 |

### 4.5 语音方案

**推荐：预录制音频文件（非实时TTS）**

| 方案 | 延迟 | 自然度 | 离线可用 | 推荐 |
|------|------|--------|---------|------|
| 预录制音频 | ~50ms | 高（可请专业配音） | ✅ 是 | ✅ 推荐 |
| 微信TTS插件 | ~200ms | 中 | ❌ 需网络 | 备选 |
| 云端TTS API | ~500ms | 高 | ❌ 需网络 | 不推荐 |

预录制音频清单（共约10条）：
- `detecting.mp3` — "正在识别"
- `green.mp3` — "绿灯，可以通行"
- `red.mp3` — "红灯，请等待"
- `yellow.mp3` — "黄灯，请注意"
- `uncertain.mp3` — "无法确认，请小心通过"
- `red_to_green.mp3` — "已变绿灯，可以通行"
- `green_to_red.mp3` — "注意，已变红灯"
- `welcome_1.mp3` — "欢迎使用绿灯行"
- `welcome_2.mp3` — "过马路时，请将手机摄像头对准前方红绿灯"
- `welcome_3.mp3` — "按住屏幕中央的大按钮开始识别"
- `disclaimer.mp3` — "本工具仅辅助判断，请结合环境确认安全"

## 6. 风险与应对

| 风险 | 影响 | 应对 |
|------|------|------|
| AI识别准确率不够 | 误报红绿灯状态，可能导致危险 | 置信度阈值策略+不确定时提示用户确认+免责声明 |
| 手机摄像头角度问题 | 用户手持不稳，拍不到红绿灯 | 引导页教用户正确持握姿势+持续识别而非单次拍照 |
| 强光/夜间识别差 | 特定条件下完全失效 | 引导语提示"如无法识别，请寻求路人帮助" |
| 法律责任 | 用户因误识别发生事故 | 免责声明+产品责任保险+明确"辅助工具"定位 |
| 微信小程序审核 | 功能可能被要求补充资质 | 提前了解审核要求，准备相关材料 |
| **摄像头权限被拒** | **用户拒绝授权摄像头，核心功能完全不可用** | **首次打开时引导授权；被拒后提供清晰的重新授权入口；语音引导用户去设置中开启权限** |
| **网络依赖（云端推理）** | **无网络时无法使用** | **引导语提示"需要网络连接"；后续V2版本支持端侧离线推理** |
